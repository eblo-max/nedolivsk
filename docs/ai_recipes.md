# «Тайные рецепты» — ИИ-генерация блюд из ингредиентов

Профессиональная концепция: игрок экспериментирует с ингредиентами, ИИ (Claude)
придумывает уникальное блюдо — название, лор и НАБОР эффектов, — а игровой код
назначает числа из детерминированного бюджета. Рецепт становится личным секретом,
который можно охранять, украсть или слить. Статус: **предложение** (не реализовано).

---

## 1. Резюме

У игры есть крафт блюд по фиксированным рецептам (`production.GOODS`, эффекты в
`balance.FLASK_EFFECTS`). «Тайные рецепты» добавляют **эмерджентный** слой: игрок
комбинирует ингредиенты из погреба/склада, и на новую комбинацию Claude
генерирует **уникальное** блюдо. Ключевой инженерный принцип — **ИИ придумывает
флейвор и ВЫБИРАЕТ эффекты, а код назначает их величину из бюджета** (как
итемизация шмота). Так вкус придумывает модель, а баланс держит код: показ=действие
и целостность экономики не ломаются. Рецепт детерминирован по хэшу ингредиентов —
одинаковое комбо всегда даёт одно и то же блюдо, генерится один раз на весь мир и
кэшируется. Это делает рецепт стабильной сущностью, вокруг которой строится
мета-игра секретов: изобрёл → владеешь эксклюзивно → могут украсть/слить.

## 2. Проблема / зачем

- Крафт статичен: конечный список рецептов, у всех одинаковый. Нет открытий,
  нет личного, нет причины экспериментировать.
- Биржа полупустая (реальная проба прода): игрокам не хватает причин активно
  торговать ингредиентами. Спрос на редкие ингредиенты под тайные рецепты оживил
  бы её.
- Уникальный для медиума крючок: живой городок в чате + ИИ = «у каждого свой
  секретный фирменный рецепт», которого больше нет ни у кого.

## 3. Видение / фантазия игрока

«Я — повар со **своим** секретом». Игрок смешивает ингредиенты, ИИ рождает блюдо с
характером («Похмельный борщ боярина Твердислава»), дающее реальный бафф. Пока
секрет твой — у тебя эдж. Но рецепт может **утечь**: вор украдёт, пьяный повар
проболтается, а самые ходовые со временем «расходятся по трактам». Ты патентуешь,
охраняешь, или сам крадёшь чужой хит и реверсишь.

## 4. Ядро механики

### 4.1 Ингредиенты — фиксированный словарь (не свободный ввод)

Экспериментируют **только** существующими ключами игры: сырьё (`balance.RESOURCES`)
+ базовые товары (`production.GOODS`). Свободного пользовательского текста в ИИ не
уходит — это снимает риск инъекций и даёт конечное пространство комбинаций (важно
для кэша и детерминизма). Комбо: 2–4 ингредиента.

### 4.2 Бюджет рецепта (числа — из кода, не из ИИ)

Бюджет в **урон-эквиваленте** (единая шкала эффектов игры) считается детерминированно
из ценности ингредиентов — тот же паттерн, что `items.item_budget_target`:

```
budget = round(BASE + Σ(ценность ингредиента) × RECIPE_BUDGET_K)
```

где ценность = `RESOURCE_PRICE[k]` для сырья или `GOODS[k].price` для товаров.
Калибровка так, чтобы обычный тайный рецепт ≈ лучшему обычному блюду (kebab +9
урона), а на редких/дорогих ингредиентах дотягивал к эксклюзиву зодчих (~×2). Никакой
рецепт не может превысить бюджет — потолок жёсткий.

### 4.3 Белый список эффектов — БАЗОВЫЙ боевой набор (пространство = FLASK_EFFECTS)

> Это стартовый набор Ф0 (быстро внедряется, всё уже посчитано в боёвке).
> Расширение словаря за пределы боя — критично для уникальности; вынесено
> в §4.6, где и лежит настоящее пространство эффектов.

Боевые эффекты — ровно те, что уже есть в бою: `dmg`, `crit` (%), `dodge` (%),
`hp`, `antidote` (bool). Конверсия в урон-эквивалент для бюджета — по существующим
константам (показ=действие с боёвкой):

| Эффект | Стоимость в бюджете |
|---|---|
| `dmg` +1 | 1 |
| `hp` +1 | 1 / `RAID_HP_TO_DMG` (=1/3) |
| `dodge` +1% | 1 / `RAID_DODGE_TO_DMG` (=1/4) |
| `crit` +1% | ≈ 0.8 (калибровка по wine/nalivka) |
| `antidote` | фикс. ~4 (ситуативно ценно) |

### 4.4 Роль ИИ vs код (сердце дизайна)

| Делает **Claude** | Делает **код** |
|---|---|
| Название (тематичное, недоливский тон) | Итоговые ЧИСЛА эффектов |
| Короткий лор/описание | Кламп суммы весов ≤ `budget` |
| ВЫБОР, какие эффекты вяжутся с ингредиентами (из белого списка) | Отсев чужих эффектов вне списка |
| Черновое распределение бюджета (веса) | Нормализация, округление, модерация текста |

Модель возвращает **структурированный JSON** (не прозу). **Важно:** JSON-схема
структурированного вывода не поддерживает числовые ограничения (`minimum`/`maximum`)
— поэтому потолок бюджета обязан жить в КОДЕ (кламп), а не в схеме. Это и есть
гарантия баланса: что бы модель ни навыдумывала, мощность назначает код.

### 4.5 Детерминизм и кэш

`combo_hash = sha1(sorted(ingredients))`. Новая комбинация → один вызов ИИ,
результат в таблицу `recipes`. Все, кто открыл то же комбо, получают готовое из БД.
Следствия: рецепт — стабильная сущность (можно украсть конкретный), «реролл ради
статов» невозможен (то же комбо = тот же рецепт, а числа и так из бюджета), расходы
на ИИ конечны.

## 4.6 Пространство эффектов и уникальность (главная правка дизайна)

**Честный диагноз.** Ингредиентов достаточно: 41 добываемых/крафтовых → **112 750**
комбинаций из 2–4 (820 + 10 660 + 101 270). Названий и лора хватит на все —
каждое комбо получит своё имя. Но при 5 БОЕВЫХ эффектах и одном эффекте на рецепт
механически рецепты **схлопываются** в ~5 семейств: разные названия, одинаковая
суть («ещё +урон»). Уникальным остаётся только текст. Ниже — 4 правки, которые
делают рецепты по-настоящему разными НА ДЕЛЕ, а не только на вид.

### 4.6.1 Расширенный словарь эффектов (за пределы боя)

Рецепт может влиять не только на рейд. Подключаем эффекты ко ВСЕМ уже живущим
подсистемам через их существующие котировки (показ=действие сохраняется — каждый
эффект читается там же, где считается число):

| Эффект | Куда цепляется (существующая котировка) | Ед. |
|---|---|---|
| `dmg` `crit` `dodge` `hp` `antidote` | рейд — `raid.flask_mods` | боевые |
| `income_pct` | гостевой сбыт — `logic.retail_total` | +% выручки |
| `yield_pct` | сбор ресурсов — множитель добычи (как «Хмельные сады») | +% сбора |
| `luck` | экспедиции — шанс «фарта»/редкой находки | +% |
| `craft_speed` | варка — сокращение времени крафта | −% времени |
| `spoilage_resist` | склад/порча — медленнее портится | +% срока |
| `guest_draw` | приток гостей — множитель посещаемости | +% гостей |
| `rep_pct` | репутация со сделки — быстрее к рангам славы | +% репы |
| `heal_pct` | вне боя — восстановление между рейдами | +% |
| `exertion_cut` | усталость персонажа — дешевле «выносливость» | −% |

Словарь: **было 5 → стало 14**. Стоимость каждого нового эффекта в бюджете
калибруется в тех же «урон-эквивалентных» единицах (напр. `income_pct` +1% ≈
ценность одной средней сделки в урон-эквиваленте), как `RAID_HP_TO_DMG`. **Цена
интеграции честная:** каждый небоевой эффект требует читать `effect_of(recipe)` в
своей котировке (retail/gather/expedition/craft) — это не «одна функция», а 4–5
точек подключения. Зато один словарь эффектов кормит все системы разом.

### 4.6.2 Мульти-эффектные профили («билды»)

Рецепт несёт **1–3 эффекта**, а не один. Бюджет делится между ними → появляются
«билды»: чистый ДД (весь бюджет в `dmg`), танк (`hp`+`dodge`), купеческий
(`income_pct`+`guest_draw`), добытчик (`yield_pct`+`luck`), гибрид (`crit`+`dmg`+
`antidote`). Код по-прежнему клампит СУММУ весов ≤ `budget` — мощность жёстко
ограничена, но её раскладка становится осмысленным выбором. ИИ предлагает раскладку
(веса), код нормализует к бюджету.

### 4.6.3 Теги свойств ингредиентов → смещение выбора эффектов

Чтобы `перец+сало` и `мёд+ягоды` давали РАЗНЫЕ по духу рецепты (а не «оба +урон»),
у каждого ингредиента — теги вкуса/свойства, которые СМЕЩАЮТ выбор ИИ (не жёстко
задают — модель всё ещё выбирает, но в рамках тематики):

| Тег | Тяготеет к эффектам | Примеры ингредиентов |
|---|---|---|
| острое | `dmg`, `crit` | перец, хрен, горчица |
| жирное/сытное | `hp`, `heal_pct` | сало, мясо дичи, молоко |
| травяное | `antidote`, `heal_pct` | травы, коренья |
| хмельное/бродящее | `crit`, `luck` | хмель, брага, вино |
| сладкое | `income_pct`, `guest_draw` | мёд, ягоды |
| солёное/копчёное | `spoilage_resist`, `income_pct` | соль, рыба |
| землистое/рудное | `yield_pct`, `exertion_cut` | руда, глина, камень |

Теги идут в промпт как контекст флейвора — «эти ингредиенты острые и хмельные,
логичный эффект — урон и крит». Так название, лор И механика вытекают из одних и
тех же ингредиентов и согласованы между собой.

### 4.6.4 Ярусы по редкости (бюджетные полосы)

Бюджет уже растёт от ценности ингредиентов (§4.2). Формализуем 4 полосы силы →
редкие/дорогие комбо дают ощутимо более мощные рецепты, а не «те же +2»:

| Ярус | Бюджет (усл.) | Типичные ингредиенты | Число активных эффектов |
|---|---|---|---|
| Обычный | ~9 | базовое сырьё | 1 |
| Необычный | ~14 | + крафт-товар | 1–2 |
| Редкий | ~20 | дорогое сырьё/товары | 2 |
| Экзотический | ~28 | эксклюзив/дичь/руда | 2–3 |

Ярус — не отдельная механика, а следствие бюджета; но он даёт игроку внятную цель:
охотиться за дорогими комбо ради 3-эффектных экзотик-билдов.

### 4.6.5 Пересчёт уникальности (числа, не на глаз)

При 14 эффектах и 1–3 на рецепт:

| Слой | Формула | Кол-во |
|---|---|---|
| Флейвор-уникальные (имя/лор) | C(41,2..4) | **112 750** комбо |
| Наборы эффектов (какие эффекты) | C(14,1)+C(14,2)+C(14,3) = 14+91+364 | **469** наборов |
| Различимые механические профили | 469 наборов × раскладки магнитуд × 4 яруса | **≈ 26 900** |

Раскладка: 1-эфф = 14×5×4 = 280; 2-эфф = 91×9×4 = 3 276; 3-эфф = 364×16×4 = 23 296.

**Итог: с ~20 механически-разных профилей (5 боевых × ~4 магнитуды, 1 на рецепт)
до ≈ 26 900 — рост примерно в ~1340×.** При этом имя и лор остаются уникальными на
все 112 750 комбо. Именно расширение словаря (4.6.1) + мульти-эффект (4.6.2) —
рычаг уникальности; ингредиенты и так были в избытке.

> Приоритет внедрения: Ф0 стартует на 5 боевых эффектах + 1 на рецепт (быстро,
> всё посчитано). Ф1 = §4.6.1–4.6.2 (словарь + мульти-эффект) — это и есть скачок
> уникальности. §4.6.3–4.6.4 (теги, ярусы) — полировка тематики и прогрессии.

## 5. Секрет → утечка → кража (мета)

- **Владение** — через существующий механизм: ключ рецепта в `story['artel']
  ['recipes']`, гейт варки по `production.owns_recipe`. Изобрёл — добавили ключ,
  ты первооткрыватель и (пока) эксклюзивный владелец.
- **Утечка** каналами: вор-фракция крадёт рецепт у соседа; «пьяный повар»
  проболтался (случайное событие story-движка); ходовой рецепт со временем
  «расходится» (эксклюзив спадает через N дней). Утечка = ключ появляется у другого
  игрока / теряет эксклюзивность.
- **Реверс** — потратить ингредиенты и попытку, чтобы «распробовать» чужой хит.

Это переиспользует фракции (воры), story-движок (события), артель-рецепты
(владение) — новых систем почти не требует.

## 6. Технический дизайн

### 6.1 Модель данных

Новая таблица `recipes` (аддитивная миграция, как `wonders`/`fame_rank`):

| Поле | Тип | Смысл |
|---|---|---|
| `combo_hash` | str, PK/uniq | ключ комбинации (детерминизм) |
| `key` | str | игровой ключ рецепта (для owns_recipe/варки) |
| `name` | str | название от ИИ (модерировано) |
| `lore` | str | описание от ИИ |
| `effects` | JSONB | итоговые числа ПОСЛЕ клампа `{dmg,crit,dodge,hp,antidote}` |
| `budget` | int | бюджет, из которого назначено (аудит) |
| `discoverer_id` | bigint | первооткрыватель |
| `status` | str | `secret` / `leaked` (эксклюзив/общий) |
| `created_at` | ts | |

### 6.2 Модуль `bot/game/ai_recipe.py` (вызов Claude)

Стек проекта — async (aiogram+aiohttp, один процесс). Используем официальный SDK
`anthropic`, асинхронный клиент с aiohttp-бэкендом:

```python
from anthropic import AsyncAnthropic, DefaultAioHttpClient
from pydantic import BaseModel

# ANTHROPIC_API_KEY из Railway-env; клиент создаём один раз на процесс
_client = AsyncAnthropic(http_client=DefaultAioHttpClient())

ALLOWED = ["dmg", "crit", "dodge", "hp", "antidote"]

class AIRecipe(BaseModel):
    name: str
    lore: str
    effects: dict[str, int]        # ВЕСА-предложение; код переназначит по бюджету

SYSTEM = """<правила + ПОЛНЫЙ каталог ингредиентов с флейвором + 3-4 few-shot
примера + тон/гардрейлы>. Верни JSON name/lore/effects; effects — только из
белого списка; тон — средневековый недоливский, без пошлости и жестокости."""

async def invent(ingredients: list[str], budget: int) -> AIRecipe | None:
    try:
        r = await _client.messages.parse(
            model="claude-haiku-4-5",          # дёшево/быстро; структ.вывод поддержан
            max_tokens=500,                    # мелкий вывод — стрим не нужен
            system=[{"type": "text", "text": SYSTEM,
                     "cache_control": {"type": "ephemeral"}}],   # кэш стабильного префикса
            messages=[{"role": "user",
                       "content": f"Ингредиенты: {ingredients}. Бюджет очков: {budget}."}],
            output_format=AIRecipe,
        )
        if r.stop_reason == "refusal":         # классификатор отказал → фолбэк
            return None
        return r.parsed_output
    except anthropic.APIError:                 # 429/5xx/сеть — SDK уже ретраит; тут фолбэк
        return None
```

Замечания (сверено по актуальному Claude API):
- **Модель** — `claude-haiku-4-5` ($1/$5 за 1M): осознанный даунгрейд с дефолтного
  Opus под массовую простую задачу «флейвор+распределение». Структурированный вывод
  Haiku 4.5 поддерживает.
- **Кэш промпта**: стабильный `system` (правила+каталог+примеры) под `cache_control`
  → повторные вызовы читают его за ~0.1× цены. У Haiku минимум кэшируемого префикса
  **4096 токенов** — правила+каталог обычно дотягивают; если нет, добить примерами.
- **`max_tokens=500`**, стрим не нужен (вывод крошечный, таймаута нет).
- **Ошибки/отказы**: `stop_reason=="refusal"` и `APIError` → возвращаем `None`,
  вызывающий уходит в фолбэк. SDK сам ретраит 429/5xx.
- **Bulk-предгенерация** (опц.): популярные комбо можно прогреть через Batches API
  (−50% цены), чтобы у игроков не было задержки на первое открытие.

### 6.3 Кламп-функция (единственный источник чисел)

```python
def assign_effects(ai_effects: dict, budget: int) -> dict:
    """ИИ предложил, КОД назначает: только белый список, сумма весов ≤ budget."""
    out = {}
    spent = 0
    for k in ("dmg", "hp", "dodge", "crit", "antidote"):     # приоритет распределения
        if k not in ai_effects:
            continue
        cost1 = EFFECT_COST[k]                                # стоимость 1 ед. в бюджете
        room = budget - spent
        val = min(int(ai_effects[k]), int(room / cost1))
        if k == "antidote":
            val = 1 if (ai_effects[k] and room >= EFFECT_COST["antidote"]) else 0
        if val > 0:
            out[k] = val
            spent += val * cost1
    return out            # гарантированно ≤ budget, только валидные эффекты
```

### 6.4 Интеграция

- **Эндпоинт** `/api/recipe/experiment` (webapi): под локом игрока — проверить
  кулдаун, списать ингредиенты, посчитать `budget`, `combo_hash`; если новый —
  `invent()` (+фолбэк), `assign_effects`, сохранить в `recipes`, добавить ключ в
  `owns_recipe`; если известен — «переоткрыл» или провал (если чужой секрет).
- **Варка/применение**: рецепт варится существующим крафтом, гейт по `owns_recipe`
  (уже есть). Эффекты применяются как у статики — см. seam ниже.
- **Показ=действие seam**: `flask_mods`/`flask_label` читают эффекты через единый
  резолвер `effect_of(key)`: сначала статический `FLASK_EFFECTS`, затем таблица
  `recipes` (кэш в памяти). Одна котировка — метка на экране == применённый эффект.

### 6.5 Кэш / стоимость / латентность

- **Двойной кэш**: глобальный combo-кэш (один вызов на комбо на весь мир) + кэш
  промпта (−90% на префиксе). Суммарный расход — считанные центы на сотни рецептов.
- **Латентность**: Haiku быстрый; первое открытие комбо — ~1–2 c (можно показать
  «повар пробует…»); повторные — мгновенно из БД.

## 7. Безопасность и анти-абьюз

- **Инъекций нет** — ингредиенты фиксированный словарь, не свободный текст.
- **Числа никогда не от ИИ** — только `assign_effects` (кламп бюджетом). Схема
  структурированного вывода числовые лимиты не enforce'ит — потому лимит в коде.
- **Модерация текста** — гардрейлы в system-промпте + код-сайд: лимит длины
  name/lore, HTML-escape (выводится в чат/мини-апп), опц. блок-лист.
- **Кулдаун + списание ингредиентов** на эксперимент — от спама API и от фарма.
- **Детерминизм = нет реролла** ради статов (то же комбо = тот же рецепт; числа и
  так из бюджета).
- **Фолбэк без падения**: ИИ недоступен/отказ → детерминированное процедурное имя +
  бюджетный эффект без лора («рецепт дорабатывается»); игра работает.
- **Ключ API** — из Railway-env, никогда в коде/логах/ответах.
- **Стоимость под контролем** — combo-кэш + prompt-кэш + Haiku; при желании
  дневной денежный потолок (счётчик вызовов).

## 8. Показ=действие

Эффекты рецепта хранятся числами и читаются той же `effect_of/flask_mods`, что и
статические блюда. Метка «+9 урона», показанная игроку, — ровно то, что применится в
бою. Бюджет виден заранее («на этих ингредиентах повар выжмет ~9 очков силы») —
рандома в мощности нет, есть творчество в НАБОРЕ и флейворе.

## 9. Экономическая целостность

- Потолок бюджета → имбы быть не может, вне зависимости от ответа ИИ.
- Эксперимент **тратит** ингредиенты → сток + спрос на бирже (оживляет её).
- Секреты/эксклюзив добавляют глубины без инфляции (мощность конечна).

## 10. Тест-план

Тесты **без реальных вызовов API** (мок клиента) — CI не ходит в сеть:
- `assign_effects` НИКОГДА не превышает бюджет (свойство на рандомных входах).
- Только эффекты из белого списка проходят; чужие ключи отсеяны.
- Детерминизм: одинаковое комбо → одинаковый `combo_hash`.
- Показ=действие: `effect_of(recipe)` == сохранённым числам; `flask_mods` считает их
  как статику.
- Фолбаковый путь (ИИ вернул `None`) не падает, даёт валидный бюджетный рецепт.
- Инъекций нет: `invent` получает только ключи из словаря (гейт на входе).
- Модерация: длинные/пустые name/lore обрезаются/эскейпятся.

## 11. План по фазам

- **Ф0 — ядро (MVP):** словарь ингредиентов, бюджет, `ai_recipe.invent` + кламп +
  фолбэк, таблица `recipes`, эндпоинт эксперимента, варка по владению, seam
  показ=действие, тесты (с моком). Полностью играбельно, без утечек.
- **Ф1 — секреты:** эксклюзив первооткрывателя, каналы утечки (воры/пьяный
  повар/время), реверс чужого рецепта.
- **Ф2 — полировка:** доска слухов о «ходовых» рецептах, обмен рецептами между
  игроками, коллекция «кулинарной книги», ачивки.

## 12. Риски и метрики

| Риск | Митигация |
|---|---|
| Имба-рецепт | жёсткий кламп бюджетом; числа только из кода |
| Стоимость ИИ | combo-кэш + prompt-кэш + Haiku + опц. дневной потолок |
| Неуместный текст | гардрейлы промпта + код-модерация + escape |
| Латентность первого открытия | «повар пробует…», опц. Batches-предгенерация популярных комбо |
| Спам эксперимента | кулдаун + списание ингредиентов |

**Метрики успеха:** доля игроков, открывших ≥1 рецепт; оборот ингредиентов на бирже
до/после; D7/D30-ретенция «поваров»; частота событий утечки/кражи (соц-вовлечение).
